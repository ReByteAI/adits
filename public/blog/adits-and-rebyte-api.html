How Adits uses the Rebyte API

One project, one VM. One Adit, one task. Files live on the VM's filesystem. Here's exactly how the hosted backend is wired.

Adits ships in two backends. Self-host runs your project in a folder under ~/.adits/projects/. Hosted runs it in a per-project VM provisioned through Rebyte's Agent Computer API. The surface is identical; the seam is what differs.

This post walks through the hosted side. If you only ever run self-host, none of the wiring below applies — the local backend collapses every concept here into "a folder and a child process."

The mapping

The hosted architecture is small enough to fit in one diagram:

Adits project   ←→   Rebyte agent computer (1 persistent VM)
       │
       ├── User submits a prompt  →  POST /v1/tasks  (one Rebyte task)
       ├── User submits again      →  POST /v1/tasks  (another task)
       └── ...

Three rules govern everything:

  • 1 project = 1 agent computer. Project creation provisions a Rebyte VM via POST /v1/agent-computers {name}. The VM is named, persistent, and lives until the user deletes the project.
  • 1 "Adit" = 1 task. Every prompt the user submits creates one Rebyte task on that VM via POST /v1/tasks. No batching. No follow-up chains. Every submit is independent.
  • The VM's filesystem is the project's state. The directory /code inside the VM is where all of the project's files live. Adits has no separate file storage — no S3 bucket, no R2, no separate document store. The VM is the storage.

The file server

If the VM holds the files, the browser needs a way to fetch them. We solve this with a small Go binary called adits-file-server that runs inside every project's VM, bound to :8080, serving /code as static HTTP.

Every Adits project gets a per-project URL of the form https://8080-<sandboxId>.<env>.rebyte.app. That's the URL the browser uses for:

  • Iframe page rendering. Open an HTML page in the bench → the iframe's src is a URL on this domain. The page loads directly from the VM, the same way it would in a standalone tab.
  • File downloads. Image previews, PDF blobs, audio streams — all fetched against the same per-project origin.
  • The bridge script. The file server splices a <script src="/_adits/inject.js" defer> tag into every HTML response, so each rendered page can talk back to the Adits host via postMessage (the Edit / Comment / Present lanes).

The file server itself is installed once per project — Adits writes a systemd unit on the VM, the unit auto-restarts across pause/resume, and we never touch it again.

Sandboxes are Lambdas

Rebyte VMs auto-pause when idle and auto-resume on first request. The whole architecture treats them like Lambdas: we never pre-warm. No keepalive loop, no /ensure-sandbox round-trip before rendering, no separate "wake the VM" step in the UI.

What that means in practice: the iframe URL is safe to embed before any task runs. The very first request to https://8080-<sandboxId>.prod.rebyte.app/some.html hits the sandbox gateway, which auto-resumes the VM and forwards the request. The user sees a brief load, then the page. The Adits server never has to babysit the VM lifecycle.

The same applies to @eng0/sdk calls (the Sandbox SDK). When Adits needs to list() files, write a new file from an upload, or delete a project's bytes, it calls the SDK directly — auto-resume happens transparently inside the SDK's gRPC client.

Webhooks for task events

Tasks run asynchronously on the VM. The agent might take 5 seconds or 5 minutes. Rather than poll, Adits uses Rebyte's webhook API:

POST /v1/webhooks
{
  "url": "https://adits.ai/api/app/webhooks/rebyte",
  "events": ["task.frame", "task.terminal"],
  "secret": "..."
}

Per-user webhook, registered the first time a user signs in. Rebyte POSTs JSON to our endpoint every time the agent emits a frame (a chunk of streamed output) or hits a terminal state (succeeded / failed / cancelled). The Adits server forwards those into Postgres + Redis pub/sub, and the browser receives them via a single SSE per active prompt.

That single-SSE-per-prompt model is what lets the chat panel update incrementally — every line the agent prints, every status change — without the client having to know which VM is running it or whether the VM is awake.

Three credentials, three layers

The hosted backend stacks three credentials. Confusing them is the most common source of "everything's broken" after a config change, so it's worth being explicit about what each one does:

LayerStored inUsed for
Clerk JWT browser (Clerk SDK) Authorization: Bearer <jwt> on calls to /api/app/*
Per-user Rebyte API key Postgres: users.rebyte_api_key API_KEY header on calls to the Rebyte relay (/agent-computers, /tasks, /webhooks)
Per-user Sandbox SDK key Postgres: users.sandbox_api_key @eng0/sdk calls against the user's sandbox base URL

The Clerk JWT is the user's identity in the browser. The Rebyte key is the user's identity inside the Rebyte org — auto-provisioned on first login via a partner key. The Sandbox SDK key is fetched once via the Rebyte relay's /sandbox/api-key route and cached on the user row; it doesn't expire, so we never refresh.

Why the local mode collapses all this

None of the above exists in self-host mode. Setting ADITS_BACKEND=local swaps three modules behind a single seam:

  • FileStore — instead of "INSERT a row in Postgres pointing at a Rebyte VM", just mkdir ~/.adits/projects/<id>.
  • FileServer — instead of the per-project Rebyte URL, adits-file-server runs as a child process of the Node server, serving the local projects directory.
  • TaskRunner — instead of POST /v1/tasks, spawn the local CLI binary (claude, codex, or gemini) with the prompt as a one-shot.

Same surface, no infrastructure. The bench, the chat, the file viewers — none of them know which backend is wired up. Both code paths satisfy the same TypeScript interface. That's the whole point: hosted is a deployment choice, not a fork.

Where to look in the code

Everything above lives under server/backend/ in the OSS repo:

  • backend/index.ts — backend selection (rebyte vs local)
  • backend/file-server.ts — the seam for "give me a project's file URL"
  • backend/rebyte/ — hosted-mode implementations: agent-computer provisioning, sandbox pool, file-server install, webhook flow
  • backend/local/ — self-host implementations: filesystem store, child-process task runner, local file-server child

If you're curious about the internals, that's the path through the codebase. Source: github.com/ReByteAI/adits.
